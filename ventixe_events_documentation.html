University of Sargodha (UOS)

Campus Event Management & Crowd Prediction System

A centralized web-based platform (Ventixe Events) for managing campus events, student registrations, ticketing, crowd prediction, and feedback, developed as a Software Engineering semester project.

Course: Software Engineering Group: 8 (Campus Event Management & Crowd Prediction System) Domain: Web Application · Event Management · AI/ML-based Prediction

1. Problem Statement

Why this system is needed

Universities organize a large number of events throughout the academic year – seminars, workshops, sports events, society activities, orientations, job fairs, and cultural programs. In many campuses these events are still managed manually or with scattered tools (WhatsApp groups, posters, spreadsheets, separate Google forms). As a result, there is no single source of truth for upcoming events, registrations, venues, or expected crowd size.

This leads to multiple practical problems for both administration and students:

  • Event information is spread across different channels, so many students miss relevant events or receive updates very late.
  • There is no centralized registration & attendance tracking, which makes it hard to know how many students actually attended a particular event.
  • Event organizers cannot predict the expected crowd, so they may under-estimate or over-estimate required resources (venue size, chairs, refreshments, security, etc.).
  • Without automated venue conflict detection, two or more events may accidentally be scheduled in the same hall at overlapping times.
  • Students have no unified event calendar and cannot easily view all events relevant to them in one place.
  • Feedback about events is often collected informally, so no structured event quality analysis is available for improvement.

The lack of a centralized, intelligent event management system causes poor coordination, resource wastage, and a weak student experience. There is also no data-driven mechanism to predict and manage crowds at events.

Problem: UOS currently lacks a unified digital platform that can manage all campus events end-to-end (creation, registration, ticketing, venue allocation, attendance tracking) and intelligently predict expected crowd size so that administrators can plan resources and avoid conflicts.

2. Objectives

What the project aims to achieve

The main objective of the Campus Event Management & Crowd Prediction System (Ventixe Events) is to design and implement a centralized campus event portal that streamlines the entire lifecycle of campus events and provides basic AI/ML-based crowd prediction support.

Specific objectives include:

  • Provide a single web-based platform where students can discover, search, and register for all official campus events.
  • Enable authorized users (organizers, staff, admins) to create, update, cancel, and manage events with proper categories, capacities, schedules, and venues.
  • Implement student registration and ticket generation with QR codes to support attendance tracking at the venue gate.
  • Design a simple crowd prediction module that uses historical booking/attendance data to estimate the expected number of attendees for upcoming events.
  • Provide a notification and reminder system to send confirmations and reminders (e.g., email) to registered students.
  • Detect venue conflicts when two events are planned for the same venue and time, and warn organizers before saving the event.
  • Collect structured feedback and ratings from attendees after events to measure event quality and support future improvements.
  • Offer basic reporting and analytics (event statistics, registrations, revenue, feedback summaries) for administrators.

3. Scope

What is included and excluded

This project covers the complete workflow for managing campus events at the University of Sargodha from the perspective of students, organizers, staff, managers, and administrators. The scope is defined in terms of included and excluded features to keep the system realistic for a semester project while still addressing the main problem.

In-Scope Functionality

  • Web-based portal for listing all campus events with categories, dates, venues, and details.
  • Role-based user accounts (student, organizer, staff, manager, admin) with authentication and basic profile management.
  • Event creation, editing, deletion, status management (draft, active, completed, cancelled, postponed).
  • Online student event registration and booking with capacity checks and duplicate registration prevention.
  • Ticket management: automatic ticket and QR code generation for confirmed registrations.
  • Basic crowd prediction based on historical booking/attendance data.
  • Notification and reminder flows for important actions (registration confirmation, event reminders – subject to technical limitations).
  • Venue information storage and simple venue conflict detection at event creation time.
  • Feedback and rating collection for completed events, with summary analytics for admins.
  • Admin dashboard with key statistics and overview of events, bookings, and financial metrics.

Out-of-Scope / Limitations

  • Full-scale production-grade AI/ML model training and deployment (a simplified prototype prediction mechanism is used instead).
  • Integration with external payment gateways (online payments can be simulated or treated as free events for this SE project).
  • Native mobile applications (only a responsive web interface is targeted).
  • Complex real-time IoT-based crowd monitoring (e.g., sensors, cameras). Only data from the web system itself is considered.
  • Enterprise-level multi-campus support; the system is designed primarily for one campus (UOS).
Scope Summary: The system delivers an end-to-end campus event lifecycle (from announcement to feedback) plus a basic ML-powered crowd prediction feature, implemented as a web application suitable for a Software Engineering semester project.

4. Functional Requirements

What the system shall do

This section summarizes the main functional requirements specified in the separate Software Requirements Specification (SRS) document for the Campus Event Management & Crowd Prediction System. Requirements are grouped by module.

4.1 Event Management Module

  • System shall allow authorized users (admin, manager, organizer) to create, edit, and delete events with all required details (title, description, dates, category, capacity, venue, image, etc.).
  • System shall support event status management (draft, active, completed, cancelled, postponed) and display status to users.
  • System shall categorize events into seminars, workshops, sports, society activities, orientations, and other types.
  • System shall enforce capacity limits for each event and prevent over-booking.
  • System shall support event search and filtering by title, description, category, status, and date.
  • System shall provide multiple event listing views (grid, list, and a basic calendar view).
  • System shall display a detailed event page with full information and registration actions for students.
  • System shall highlight featured or upcoming events on the landing page.

4.2 User Management Module

  • System shall allow students to register accounts using email, password, student ID, name, department, and phone.
  • System shall provide separate login functionality for students and admins.
  • System shall verify student email addresses using a one-time verification code.
  • System shall hash and store passwords securely (bcrypt-based in the implementation).
  • System shall implement role-based access control for roles including student, admin, manager, organizer, and staff.
  • System shall manage user sessions securely and automatically expire them after inactivity.
  • System shall allow users to view their profile information (profile update may be partially available).
  • Admin shall be able to view and manage all registered users (e.g., list of students).

4.3 Registration & Booking Module

  • Students shall be able to register for events directly from the event details page (one-click registration).
  • System shall prevent duplicate registrations for the same event by the same student.
  • System shall check event capacity before confirming a registration.
  • System shall maintain booking status (pending, confirmed, cancelled) and allow admin/staff to manage bookings in an admin interface.
  • Students shall be able to view their booking history in a dedicated page.
  • Admin/staff shall be able to view and filter all bookings by event and status.

4.4 Ticket Management Module

  • For each confirmed booking, system shall automatically generate a ticket with a unique ticket code.
  • System shall generate a QR code for each ticket, encoding ticket and user information.
  • Students shall be able to view their tickets (including QR codes) from the portal.
  • System shall support download/print view of tickets (at least as a web-printable page; full PDF export may be partial).
  • System is intended to support ticket validation and check-in at the event gate (using QR scanning) to mark attendance.

4.5 Crowd Prediction Module (AI/ML Component)

  • System shall collect historical attendance/booking data for past events and store it for analysis.
  • System shall provide a mechanism to predict expected attendance for a new event using a simple ML-based or rule-based model.
  • System shall be able to analyze factors that influence attendance (e.g., event category, time, date, type).
  • System shall display predicted attendance to organizers/admins on the event management interface.
  • System aims to track prediction accuracy by comparing predicted values with actual attendance after the event.
  • System shall use prediction results to provide resource recommendations (venue size, staff, etc.) at least at a basic level.

4.6 Notification & Reminder Module

  • System shall support email notifications using an external SMTP or email service.
  • System shall be able to send registration confirmations to students after successful booking.
  • System shall send event reminders before an event (e.g., one day or one hour before start time), based on configuration.
  • System shall send update and cancellation notifications when event details are changed by organizers/admins.
  • System may support additional channels (push/SMS) conceptually, but these are implementation-dependent.

4.7 Venue Management Module

  • System shall maintain a basic venue database (halls, grounds, rooms) with information such as name, capacity, and location.
  • System shall use venue information during event creation to associate each event with a specific venue.
  • System shall perform venue conflict detection to avoid overlapping events in the same venue and time slot.
  • System is designed to support a venue availability calendar so organizers can see when a venue is free or booked.

4.8 Feedback & Rating Module

  • Students (and optionally guests) shall be able to submit feedback for events they attended.
  • System shall support 1–5 star ratings along with text comments.
  • System shall allow anonymous feedback submission for public visitors (no login required) where configured.
  • System shall provide admins with a feedback analytics view to study event quality and comments.

4.9 Reporting & Analytics Module

  • System shall provide an admin dashboard with key metrics (number of events, registrations, revenue, etc.).
  • System shall display registration trends over a recent time window.
  • System shall provide financial statistics for events (if ticket pricing is enabled).
  • System shall generate event performance reports (attendance, feedback, revenue) and basic student statistics.

4.10 Administrative Module

  • System shall provide a central admin dashboard for managing events, users, bookings, gallery, invoices, and messages.
  • Admin shall be able to configure core settings of the system (within the limitations of the project implementation).
  • Admin shall manage roles and permissions for different users.
  • System design includes concepts of audit logging and backup & restore of data for reliability (implementation may be partial).
Note: For each module above, the detailed SRS lists individual requirement IDs (e.g., FR-EV-001, FR-UM-001). This documentation provides a condensed summary suitable for project reports and presentations.

5. Non-Functional Requirements

Quality attributes of the system

Non-functional requirements describe the quality attributes of the system such as performance, security, usability, reliability, scalability, and maintainability.

5.1 Performance

  • Pages should generally load within 3 seconds under normal network conditions.
  • API endpoints should respond within approximately 2 seconds for normal workloads.
  • Database queries should be optimized using indexes on frequently accessed fields.
  • System should handle around 100 concurrent users (for university-scale usage) with connection pooling.
  • Frequent data (like event lists) should use caching and pagination to reduce server load.

5.2 Security

  • All user passwords must be encrypted (hashed) using an industry-standard algorithm (bcrypt).
  • System shall secure sessions using HTTP-only cookies and appropriate SameSite settings to mitigate hijacking.
  • All input must be validated and sanitized to prevent SQL/NoSQL injection, XSS, and other injection attacks.
  • System shall enforce role-based access control so only authorized roles can access admin/management features.
  • System design recommends CSRF protection, HTTPS enforcement in production, and security headers for deployment.

5.3 Usability

  • System shall provide a responsive UI that works on desktops, tablets, and mobile devices.
  • Navigation should be intuitive and consistent, with clear menus for students and admins.
  • Error and success messages should be clear and helpful, guiding the user to correct mistakes.
  • System should use semantic HTML and basic accessibility practices; full WCAG compliance may be a future enhancement.

5.4 Reliability

  • System shall handle errors gracefully and show user-friendly error pages.
  • System shall ensure data integrity using unique keys and validation rules
  • Database connections should be recoverable if temporary failures occur.
  • Important events and errors should be logged for debugging and auditing (at least at a basic level in this project).

5.5 Scalability

  • Application architecture should be sufficiently modular and stateless to support horizontal scaling in future deployments.
  • The database (MongoDB) is chosen to handle growing data volume with indexes and connection pooling.

5.6 Maintainability

  • Codebase should be well-organized into modules (blueprints, services, utilities) with a clear folder structure.
  • Configuration should be externalized using configuration files and environment variables.
  • Version control (Git) should be used for tracking changes.

5.7 Compatibility & Availability

  • System should work on modern web browsers (Chrome, Firefox, Edge, etc.) with responsive design for mobile browsers.
  • For the scope of the SE project, high availability is not enterprise-grade but the system should be stable and usable during academic usage and demos.
Summary: The non-functional requirements ensure that Ventixe Events is not only functionally correct but also reasonably fast, secure, usable, and maintainable for a university-scale deployment.

6. Use Cases & Use Case Diagrams

Functional view from the user perspective

This section describes who uses the system (actors), what they can do (use cases), and which UML use case diagrams have been prepared for the project. Details are based on the Use Cases Specification document.

6.1 System Actors

  • Guest / Visitor: Unauthenticated user who can browse events, view details, and submit public feedback.
  • Student: Registered user who can manage profile, register for events, view bookings and tickets, and give feedback.
  • Event Organizer: User responsible for creating and managing events, monitoring registrations, and viewing event analytics.
  • Staff Member: Operational staff who manage approvals, check-in attendees, and generate event/attendance reports.
  • Manager: Higher-level user who focuses on financial reports, venue management, and overall event statistics.
  • System Administrator (Admin): Super-user who manages users, roles, system settings, categories, and overall system health.
  • System / External Services: Automated actors such as the Email System, ML Prediction Engine, and QR Scanner.

6.2 Main Use Cases (High-Level)

The complete use case document defines over 100 detailed use cases. For overview diagrams and presentations, the following core use cases are emphasized:

  • Account & Access: Register Student Account, Login, Logout, View/Edit Profile, Verify Email, Change/Reset Password.
  • Event Discovery: View Events, Search/Filter Events, View Event Details, View Event Calendar, View Featured Events.
  • Registration & Booking: Register for Event, Cancel Registration, View My Bookings, Approve/Reject Registration, View All Registrations.
  • Ticketing: Generate Ticket, Generate QR Code, View Tickets, Download/Print Ticket, Validate Ticket, Scan QR (Check-in).
  • Crowd Prediction: Collect Historical Data, Train Prediction Model, Predict Attendance, Display Prediction, Track Prediction Accuracy, Recommend Resources.
  • Venue & Conflict Management: Add/Edit/Delete Venue, Check Venue Availability, Detect Venue Conflict, Book Venue, View Venue Calendar.
  • Notification & Reminder: Send Registration Confirmation, Send Event Reminder, Send Update/Cancellation Notification, Send Announcements.
  • Feedback & Rating: Submit Feedback, Rate Event, View Feedback, View Feedback Analytics, Moderate Feedback.
  • Reporting & Administration: View Dashboard, Generate Reports, Export Reports, Manage User Accounts, Assign Roles, Configure System Settings.
